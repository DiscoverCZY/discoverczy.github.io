Files.list引起的服务崩溃

2020-6-6 发布在 技术, 爬坑

今天，线上一个运行了半月有余的服务崩溃了。日志内大量的too many open files异常，会出现这个异常应该是服务肯定存在大量没有释放的文件资源，那就先来查看下进程的文件资源使用情况，确认一下。

ps -aux|grep service|grep -v grep|awk '{print $2}'|xargs lsof -p|wc -l

发现该服务进程下保持近8000的文件资源占用，有点夸张了。 接下来需要具体查看下到底是什么类型的文件资源被大量占用，可以在代码内方便定位。

ps -aux|grep service|grep -v grep|awk '{print $2}'|xargs lsof -p

如下所示，发现了大量DIR资源的占用未释放

COMMAND  PID   USER    FD       TYPE             DEVICE SIZE/OFF       NODE NAME
java    21582 service  rtd       DIR              252,4      335         96 /

那基本能定位到应该是对目录的某个操作，没有释放资源。开始排查服务代码中涉及目录操作的部分，发现了一段可疑的代码。

boolean isDone = Files.list(directory).anyMatch(file -> file.getFileName().equals(configuration.getFixDoneFile()));

这是服务中唯一对目录操作的代码，但是从表面看来似乎没什么问题。在常规思维内，对目录操作不是不应该涉及流打开和关闭的操作的吗？先别急着下定论，File.list这个方法是JDK提供的，我们来看看里面具体的实现。

public static Stream<Path> list(Path dir) throws IOException {
        DirectoryStream<Path> ds = Files.newDirectoryStream(dir);
        try {
            final Iterator<Path> delegate = ds.iterator();
           // Re-wrap DirectoryIteratorException to UncheckedIOException
            Iterator<Path> it = new Iterator<Path>() {
                @Override
                public boolean hasNext() {
                    try {
                        return delegate.hasNext();
                    } catch (DirectoryIteratorException e) {
                        throw new UncheckedIOException(e.getCause());
                    }
                }
                @Override
                public Path next() {
                    try {
                        return delegate.next();
                    } catch (DirectoryIteratorException e) {
                        throw new UncheckedIOException(e.getCause());
                    }
                }
            };
		return StreamSupport.stream(Spliterators.spliteratorUnknownSize(it, 	Spliterator.DISTINCT), false).onClose(asUncheckedRunnable(ds));
        } catch (Error|RuntimeException e) {
            try {
                ds.close();
            } catch (IOException ex) {
                try {
                    e.addSuppressed(ex);
                } catch (Throwable ignore) {}
            }
            throw e;
        }
    }

仔细研读Files .list的方法后，发现它对目录的操作是以打开流的方式操作的

DirectoryStream<Path> ds = Files.newDirectoryStream(dir);

然后注册了Stream的close回调方法。这意味着如果你对Stream不调用close的话，这个Files.list操作就会有文件资源没有被释放的风险。

return StreamSupport.stream(Spliterators.spliteratorUnknownSize(it,Spliterator.DISTINCT), 								false).onClose(asUncheckedRunnable(ds));

于是修改了代码如下：

Stream<Path> dir = Files.list(directory);
        boolean isDone = dir.anyMatch(file -> file.getFileName().equals(configuration.getFixDoneFile()));
        dir.close();

部署到UAT环境测试了一段时间后，再也没有发现服务进程下存在未被释放的DIR类型资源，问题解决。
